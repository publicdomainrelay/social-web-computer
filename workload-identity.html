Workload Identity Federation

A provisioned workload never holds a long-lived secret

Instead of a provider baking a static API key or password into a VM at boot, a bid can attach a compute.config.wif.simple record: a pointer to an OIDC issuer the workload can prove its own identity to, in exchange for a short-lived token scoped to exactly what it needs.

The record

issuer_uriThe OIDC issuer the workload will authenticate against.
to_issueThe audience the token should be scoped to — an API identifier plus an actx tying it to this specific contract.
subjectThe identity the token asserts — e.g. actx:<team>:plc:<requester-did>:role:<role>.
token_pathWhere the workload writes the token once it has one.
url_routeThe path on the issuer to call to redeem a proof for a token.
accept_pathWhere the signed accept bundle is written on the guest for identity proof.
 
$type: compute.config.wif.simple
accept_path: $HOME/secrets/market/accept.json
issuer_uri: https://droplet-oidc.example.com
to_issue: api://provider?actx=...
token_path: /var/run/secrets/wid/token
url_route: /v1/oidc/issue
subject: actx:<team>:plc:<requester-did>:role:<role>

This is orthogonal to policy mode: policy decides whether a workload gets scheduled at all; WIF decides what it can authenticate as once it's running.

A real deployment: fedproxy.com

The reference bidder uses this pattern to hand a freshly provisioned VM a browser-accessible terminal without ever shipping it a static password.

issuer

A small OIDC + RBAC service the provider runs

Exposes POST /v1/oidc/prove and POST /v1/oidc/issue. A VM proves custody of its SSH key by signing a challenge; the issuer replies with a short-lived, audience-scoped OIDC token.

fedproxy-client

A systemd service cloud-init installs on the guest

Runs with AUTH_PLUGIN=oidc, pointed at the issuer, and opens an outbound tunnel so the VM is reachable at a stable subdomain without any inbound port.

the payoff

A credential fetched, not shipped

Once the VM holds its scoped token, it calls the relay for its own credentials record. The relay's middleware validates the OIDC bearer before returning anything. No password was ever baked into the image.

The exchange, step by step

1

The provider attaches a wif.simple config to its bid, naming an issuer it already runs.

2

The requester's own RBAC records what that issuer is trusted to authorize, before accepting anything from this provider.

3

On accept, cloud-init writes the signed accept bundle to accept_path and starts the identity client alongside the workload.

4

The workload proves itself to issuer_uri — in the reference case, an SSH-signed challenge — and is issued a token scoped to the subject and to_issue audience.

5

The token is written to token_path and used for exactly the calls it was scoped for. It expires; nothing longer-lived was ever issued.
